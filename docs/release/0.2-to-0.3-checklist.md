# Release Readiness Checklist — 0.2.0 → 0.3.0

> All the engineering work for the strict-cutover release is committed
> in commits `80d5168..bf7ef01` on `develop`. The remaining items are
> **operational** — they require secrets and credentials that live in
> the release pipeline, not in the codebase. This document is the
> sequence the release team executes to actually ship.

## Prerequisites (CI secrets — configure before any step below)

| Secret | Where it goes | Why |
|---|---|---|
| `VERCEL_TOKEN` | GitHub Actions repository secret | Drives `.github/workflows/deploy-vercel-smoke.yml` (T4.1) |
| `NPM_TOKEN` | GitHub Actions repository secret | Drives `.github/workflows/release-coordinated.yml` (T9.1) |

For each: GitHub repo → Settings → Secrets and variables → Actions → New repository secret.

## Step 1 — Publish `theokit@0.2.0` to `latest`

```bash
# From the release machine:
# 1. Make sure develop is clean and pulled.
git checkout develop && git pull origin develop

# 2. Trigger the coordinated publish workflow.
gh workflow run release-coordinated.yml -f dist_tag=latest

# 3. Watch it run.
gh run watch
```

**Verification:**

```bash
# Smoke-install in a clean temp dir:
mkdir /tmp/smoke-0.2 && cd /tmp/smoke-0.2
npm install theokit
node -e "console.log(require('theokit/package.json').version)"
# Expected: 0.2.0
```

**Closes T9.1 ACs:**
- [x] `theokit@0.2.0` resolvable via `npm install theokit`

## Step 2 — Wait for warn-mode telemetry (4-6 weeks)

> The 0.3.0 plan explicitly mandates this window. Don't shortcut it.

Real-world deploy users running `0.2.0` will emit `csrf.warn` events
that the framework captures via the AuditLogger they configure (or the
new `/__theo/csrf-readiness` endpoint when `exposeReadinessEndpoint`
is set). Coordinate with the largest 5 deploys: have they observed any
recurring `csrf.warn` payloads in their Sentry/Datadog?

**Decision gate:** Only proceed to Step 3 when warn-mode reports have
been collected from 5+ real production deploys AND no `csrf.warn`
event was observed that would have caused a 403 in strict mode.

## Step 3 — Vercel deploy validation

```bash
# Trigger the deploy-smoke workflow against the develop branch HEAD.
gh workflow run deploy-vercel-smoke.yml
gh run watch

# Verify deploy-evidence.jsonl now has an entry with status=pass + a
# vercel.app URL.
git pull origin develop
tail -1 deploy-evidence.jsonl | grep -q "vercel.app" && echo "REAL DEPLOY RECORDED"
```

**Closes T4.1 AC:**
- [x] `deploy-evidence.jsonl` includes at least 1 real Vercel deploy

## Step 4 — Publish `theokit@0.3.0-beta.0` to `next`

```bash
# Bump version in packages/theo/package.json to 0.3.0-beta.0 (changeset
# flow is the existing mechanism; or edit manually then commit).
gh workflow run release-coordinated.yml -f dist_tag=next
gh run watch
```

**Verification:**

```bash
mkdir /tmp/smoke-0.3-beta && cd /tmp/smoke-0.3-beta
npm install theokit@next
node -e "console.log(require('theokit/package.json').version)"
# Expected: 0.3.0-beta.0
```

**Closes T9.1 ACs:**
- [x] `theokit@0.3.0-beta.0` resolvable via `npm install theokit@next`
- [x] `deploy-evidence.jsonl` includes 2 release entries (0.2.0 + 0.3.0-beta.0)

## Step 5 — One-week beta gate

> Per CLAUDE.md 0.3.0 prerequisites: "Ship `0.3.0-beta.0` to `next`
> tag first; gather feedback for at least one week; only then promote
> to `latest`."

During the week:
1. Announce on Discord: "0.3.0-beta.0 is out on `npm install theokit@next`. Run `theokit check --upgrade-readiness 0.3` against your app first."
2. Monitor GitHub issues for `0.3.0-beta` reports.
3. **Revert criteria:** any CRITICAL bug report → demote `next` tag,
   investigate, fix, re-release beta.

## Step 6 — Promote 0.3.0 to `latest`

Only after the beta gate passes with zero CRITICAL reports:

```bash
npm dist-tag add theokit@0.3.0 latest --userconfig $NPMRC
```

(The `release-coordinated.yml` workflow can also accept `dist_tag=latest`
with a 0.3.0 version bump for atomic promotion.)
