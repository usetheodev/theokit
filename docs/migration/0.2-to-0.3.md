# Migration Guide: TheoKit 0.2 → 0.3

> **0.3.0 is the strict cutover release.** CSRF defaults flip from `warn`
> to `strict` and CSP from `report-only` to `enforce`. This guide tells
> you what will break, how to detect it before you bump, and how to fix it.

## TL;DR (5 commands)

```bash
# 1. Run the static check — finds breakage before you bump.
npx theokit check --upgrade-readiness 0.3

# 2. (Production) Find every endpoint that emitted csrf.warn in your prod logs.
node -e "process.stdin.on('data',d=>d.toString().split('\\n').filter(Boolean).forEach(l=>{try{console.log(JSON.parse(l).path)}catch{}}))" < app.log | sort -u

# 3. Try the beta (no commit — installs in node_modules only).
npm install theokit@next

# 4. Run your tests against the beta.
npm test

# 5. Roll back if something broke.
npm install theokit@latest
```

If step 1 reports zero violations AND step 4 passes, you're ready for `0.3.0`.

## What changed

### 1. CSRF default: `warn` → `strict`

In 0.2.x, requests that fail CSRF (missing `X-Theo-Action: 1` header OR
mismatched `Origin`) were **allowed through** and emitted a `csrf.warn`
event. In 0.3.0, they return **403 with `CSRF_INVALID`**.

This affects:

- Custom `fetch()` calls in your app that don't use `theoFetch` or `<Form>`
- Third-party widgets posting to your `/api/*`
- Webhooks (e.g. Stripe) that don't carry a TheoKit header

### 2. CSP default: `report-only` → `enforce`

Inline `<script>` blocks without a nonce are now **blocked by the
browser**. This breaks:

- Analytics snippets (gtag, Plausible)
- Chat widgets (Intercom, Crisp)
- Any `dangerouslySetInnerHTML` containing `<script>`

### 3. `useAgentStream` auto-attaches `X-Theo-Action`

The default scaffold's chat composer was the most visible victim of #1
in 0.2.x — every send emitted a `csrf.warn`. In 0.3.0+ the hook attaches
the header on every non-GET fetch. **You do not need to do anything for
this** — it's a runtime-internal fix.

## Detection (do this BEFORE you bump)

### Static check

```bash
npx theokit check --upgrade-readiness 0.3
```

Sample output:

```
  ✗ Upgrade-readiness 0.3: 3 violation(s)

  app/checkout.tsx:14  [csrf-missing-header]
    fetch(POST) without X-Theo-Action header — will return 403 under 0.3.0 strict CSRF
    fix: Use theoFetch(path, { method, body }) OR add headers: { 'X-Theo-Action': '1' }

  public/index.html:11  [inline-script]
    Inline <script> without src — blocked under 0.3.0 enforce CSP without unsafe-inline
    fix: Move into a real script file (e.g. /main.js) and load with <script src='/main.js'>, or thread ctx.nonce

  app/widgets/intercom.tsx:8  [dangerously-set-inline-script]
    dangerouslySetInnerHTML payload contains <script> — blocked under 0.3.0 enforce CSP
    fix: Move the script into a real module loaded via <script src="...">, or thread ctx.nonce
```

Exit code 1 if any violations are found; 0 if clean. Use this in CI.

### Production log audit

If you've been running 0.2.x in production with the warn-mode log enabled,
your logs already contain a full record of every endpoint that would
break under 0.3.0. Two recipes:

**Variant A — with jq (Linux/macOS dev machines):**

```bash
grep '"event":"csrf.warn"' app.log | jq -r '.path' | sort -u
```

**Variant B — Node-only (CI / Windows / Alpine — no shell tools required):**

```bash
node -e "process.stdin.on('data',d=>d.toString().split('\\n').filter(Boolean).forEach(l=>{try{console.log(JSON.parse(l).path)}catch{}}))" < app.log | sort -u
```

Both produce the same list of paths. Variant B is what the framework's
own recipe test uses, so CI portability is guaranteed.

### In-app readiness endpoint (dev + opt-in prod)

If your app uses TheoKit dev mode OR you opt in via
`config.security.csrfTelemetry.exposeReadinessEndpoint = true`, GET
`/__theo/csrf-readiness` for a live JSON summary of warn events keyed
by route. Useful when you don't want to grep logs manually:

```bash
curl http://localhost:3000/__theo/csrf-readiness
# {
#   "generatedAt": "2026-05-21T10:00:00.000Z",
#   "totalEvents": 14,
#   "routes": [
#     { "method": "POST", "path": "/api/chat", "reason": "Missing X-Theo-Action header", "count": 9 },
#     { "method": "POST", "path": "/api/checkout", "reason": "Origin mismatch", "count": 5 }
#   ]
# }
```

To reset the counters between test runs:

```bash
curl -X POST -H "X-Theo-Action: 1" -H "Origin: http://localhost:3000" \
  http://localhost:3000/__theo/csrf-readiness/reset
```

The reset endpoint enforces CSRF on itself (X-Theo-Action + same-origin).

## Fix recipes

### Pattern A — custom `fetch()` POST

```ts
// Before
const res = await fetch('/api/checkout', {
  method: 'POST',
  body: JSON.stringify(data),
})

// After — option 1: theoFetch (recommended)
import { theoFetch } from 'theokit/client'
const res = await theoFetch('/api/checkout', { method: 'POST', body: data })

// After — option 2: keep fetch + attach the header
const res = await fetch('/api/checkout', {
  method: 'POST',
  headers: { 'X-Theo-Action': '1', 'Content-Type': 'application/json' },
  body: JSON.stringify(data),
})
```

### Pattern B — inline `<script>` in layout

```tsx
// Before
<head>
  <script dangerouslySetInnerHTML={{ __html: 'gtag("config", "GA-...")' }} />
</head>

// After — option 1: external script file
<head>
  <script src="/analytics.js" />
</head>

// After — option 2: nonce (advanced)
import { useContext } from 'react'
import { CspNonceContext } from 'theokit/client'

function Analytics() {
  const nonce = useContext(CspNonceContext)
  return <script nonce={nonce} dangerouslySetInnerHTML={{ __html: 'gtag(...)' }} />
}
```

### Pattern C — webhook endpoint (e.g. Stripe)

Webhooks don't have a browser context and can't send `X-Theo-Action`.
Opt out per-route AND verify the upstream signature:

```ts
// server/routes/webhook-stripe.ts
import { defineRoute } from 'theokit/server'
import { verifyStripeSignature } from '../lib/stripe.js'

export default defineRoute({
  csrf: false,  // opt out — we verify Stripe's HMAC signature instead
  handler: async ({ request, body }) => {
    if (!verifyStripeSignature(request.headers.get('stripe-signature'), body)) {
      return new Response('Invalid signature', { status: 401 })
    }
    // ... handle event ...
    return { ok: true }
  },
})
```

### Pattern D — `<form action>` POST

```tsx
// Before
<form action="/api/__actions/createUser" method="POST">

// After — use the framework's Form component (auto-attaches CSRF token)
import { Form } from 'theokit/client'
<Form action="createUser">
```

## Beta gate (`@next`)

The 0.3.0 line ships first to the `next` dist-tag on npm:

```bash
npm install theokit@next
```

This installs `0.3.0-beta.0` (or the latest beta) without affecting
`@latest`. Once you've validated against your app, promote to latest in
your own `package.json`:

```jsonc
{
  "dependencies": {
    // From this:
    "theokit": "^0.2.0",
    // To this (after npm publishes 0.3.0 to latest):
    "theokit": "^0.3.0"
  }
}
```

If you pin with `"theokit": "*"` you will be **auto-upgraded** when
`0.3.0` is promoted to latest. Pin a major if you want manual control.

## Rollback

If 0.3.0 breaks something you didn't catch, revert in your `package.json`:

```bash
npm install theokit@0.2 --save-exact
```

Then file an issue at https://github.com/usetheodev/theokit/issues with:

- The `theokit check --upgrade-readiness 0.3` output that DIDN'T catch your case
- The actual error / log line from your prod incident

We'll add the pattern to the static check so future migrations catch it.
