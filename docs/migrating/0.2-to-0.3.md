# Upgrading from 0.2.x to 0.3.0

TheoKit 0.3.0 flips two defaults that were shipped as warnings in 0.2.0:
the CSRF policy moves from `warn` to `strict`, and the CSP policy moves
from `report-only` to `enforce`. This guide takes you from the 0.2.x
production runtime to a clean 0.3.0 cutover without surprise 403s and
without blocked inline scripts in the browser.

If you have been reading `csrf.warn` lines in your logs since you bumped
to 0.2.0, this is the document that turns those lines into a checklist.

## TL;DR

- Two breaking defaults: **CSRF strict** (rejects non-GET without
  `X-Theo-Action: '1'` header) and **CSP enforce** (blocks inline
  `<script>` without a per-request nonce).
- The audit-and-fix loop is: (1) run
  `npx theokit check --upgrade-readiness 0.3`, (2) grep
  `"event":"csrf.warn"` across your production logs, (3) refactor each
  hit, (4) test on the npm `next` channel, (5) flip to `latest`.
- If you cannot finish before bumping, you have escape hatches:
  `defineRoute({ csrf: false })` per route,
  `config.security.csrf: 'warn'` global, or
  `config.security.headers.cspMode: 'report-only'` global. Use them as
  temporary scaffolding, not as a permanent posture.

## Prerequisites

These six items are non-negotiable. Skip any one and the cutover will
embarrass you in production.

1. **You have been running 0.2.0 in production for at least 4–6 weeks.**
   Without warn-mode telemetry, you don't know which endpoints break.
2. **`useAgentStream` is on 0.2.1 or later.** Earlier versions emit
   `csrf.warn` on every chat send. 0.3.0 turns those into 403s.
3. **You have a migration guide reader budget.** This document is ~450
   lines. Read it through before touching `theo.config.ts`.
4. **You have audited `csrf.warn` logs from production.** Not staging,
   not dev — actual production traffic for the warn-mode window.
5. **You have a per-request nonce strategy.** If your `app/layout.tsx`
   ships inline `<script>` (analytics, intercom, sentry), you need to
   either thread `ctx.nonce` to them or move them to external `src=`
   files.
6. **You ship to a beta cohort first.** `theokit@0.3.0-beta.0` lives on
   npm dist-tag `next`. Promote to `latest` only after a one-week beta
   window with zero CRITICAL reports.

## Step-by-step migration

### 1. Run the upgrade-readiness scanner

The scanner walks your `app/`, `server/`, and `public/` directories and
emits a machine-readable violations report. It does NOT modify your
source.

```bash
npx theokit check --upgrade-readiness 0.3
```

It reports three rule classes:

| Rule | What it catches |
|---|---|
| `csrf-missing-header` | raw `fetch(...)` with method POST/PUT/PATCH/DELETE missing `X-Theo-Action: '1'` |
| `inline-script` | `<script>...</script>` in `.html` files without `src=` |
| `dangerously-set-inline-script` | React `dangerouslySetInnerHTML` payload containing `<script>` |

Exit code is `1` if any violations are found (CI fails). Add
`--allow-warnings` to soften the gate while you migrate, or `--json` for
machine output.

### 2. Audit your warn logs

The 0.2.0 framework emits a structured warning every time a request that
would fail under strict CSRF arrives:

```json
{"event":"csrf.warn","method":"POST","path":"/api/login","reason":"Missing X-Theo-Action header","code":"CSRF_STRICT_CUTOVER","docsUrl":"https://theokit.dev/upgrade/csrf-strict-cutover","warnOnce":true}
```

Run this against your production logs to enumerate every endpoint that
will start failing:

```bash
grep '"event":"csrf.warn"' production.log | jq -r '.path' | sort -u
```

That gives you a deduplicated list of paths. For each one, you decide:
fix the client call site OR opt the route out (see Escape hatches).

> **Note on dedup:** the `warnOnce` field is `true`. Each unique
> `event:method:path` triple emits once per process. That keeps logs
> clean but means you must sample your logs across multiple deploys to
> see the full surface — the first request after each restart wins.

### 3. Refactor each violation site

The fix shape is uniform: swap raw `fetch` for `theoFetch`, or attach
the header explicitly.

```diff
- await fetch('/api/transfer', {
-   method: 'POST',
-   headers: { 'content-type': 'application/json' },
-   body: JSON.stringify({ amount: 100 }),
- })
+ await theoFetch('/api/transfer', {
+   method: 'POST',
+   body: { amount: 100 },
+ })
```

`theoFetch` attaches `X-Theo-Action: '1'` automatically and serializes
the body. If you need raw `fetch` (e.g., you're calling a third-party
API that just happens to live under the same origin), add the header
yourself:

```diff
  await fetch('/api/transfer', {
    method: 'POST',
-   headers: { 'content-type': 'application/json' },
+   headers: {
+     'content-type': 'application/json',
+     'X-Theo-Action': '1',
+   },
    body: JSON.stringify({ amount: 100 }),
  })
```

### 4. Audit inline scripts

Even if your CSRF flows are clean, inline `<script>` blocks in your HTML
or in `dangerouslySetInnerHTML` will be blocked under 0.3.0's enforce
CSP without `unsafe-inline`. Find them:

```bash
grep -RIn '<script' app/ public/ --include='*.html' --include='*.tsx'
```

Each hit needs one of:

1. **Move to a real module.** Save the body as `public/main.js` and
   reference it via `<script src="/main.js"></script>`. CSP allows
   external scripts.
2. **Thread `ctx.nonce`** through to the script. The framework auto-
   nonces its own SSR hydration script; user-authored inline scripts
   opt in by reading `ctx.nonce`:

   ```tsx
   export default function Layout({ ctx, children }) {
     return (
       <html>
         <head>
           <script nonce={ctx.nonce}>{`
             window.dataLayer = window.dataLayer || []
           `}</script>
         </head>
         <body>{children}</body>
       </html>
     )
   }
   ```

### 5. Switch your fetcher

If your codebase mixes `fetch` and `theoFetch`, standardize. The
project-wide grep is your friend:

```bash
grep -RIn 'fetch(' app/ src/ --include='*.ts' --include='*.tsx' \
  | grep -v 'theoFetch' \
  | grep -E "method:\s*'(POST|PUT|PATCH|DELETE)'"
```

Every hit becomes a refactor target. Re-run the upgrade-readiness
scanner after each round to track progress.

### 6. Test in `0.3.0-beta.0`

Pin to the beta dist-tag before promoting:

```bash
npm install theokit@next     # or pnpm add theokit@next
```

Run your full integration + E2E suite. The beta is API-compatible with
the GA; the only difference is the channel name.

### 7. Promote to `0.3.0`

When the beta has been clean for a week:

```bash
npm install theokit@latest   # or pnpm add theokit@latest
```

That's the cutover. Subsequent patches ship on `latest` as usual.

## CSRF Strict Cutover

This is the anchor section that `csrf.warn` payloads link at via
`docsUrl: https://theokit.dev/upgrade/csrf-strict-cutover`. Until the
hosted docs site ships in 0.4.0, the canonical content lives in this
file.

What changes: every state-mutating method (POST, PUT, PATCH, DELETE)
without `X-Theo-Action: '1'` returns HTTP 403 with body
`{"error":{"code":"CSRF_INVALID","message":"..."}}`. The Origin header
is also enforced — if the request carries `Origin` that doesn't match
`Host`, it fails the same way.

The two-line summary of how to clear every CSRF warning:

1. Use `theoFetch(path, { method, body })` from `@theokit/client`. It
   attaches the header for you.
2. For raw `fetch`, add `'X-Theo-Action': '1'` to `headers`.

See [Step 3](#3-refactor-each-violation-site) for diff examples.

## Escape hatches

Use these when you need to ship 0.3.0 before you've cleared every
violation. Treat them as temporary — flip back to defaults as soon as
you can.

### Per-route opt-out

Apply on a single route handler. Use this for endpoints that genuinely
do not need CSRF (e.g., webhooks signed by another mechanism):

```ts
// server/routes/stripe-webhook.ts
import { defineRoute } from '@theokit/core'

export default defineRoute({
  POST: {
    csrf: false, // ← skip CSRF check; signature verification is elsewhere
    handler: async ({ body }) => {
      // verify Stripe-Signature header here
      return { ok: true }
    },
  },
})
```

### Global warn-mode fallback

Reverts CSRF to 0.2.x posture for ALL routes:

```ts
// theo.config.ts
import { defineConfig } from '@theokit/core'

export default defineConfig({
  security: {
    csrf: 'warn', // ← back to warn (default in 0.3.0 is 'strict')
  },
})
```

### Global CSP report-only

Reverts CSP to 0.2.x posture. Useful when you have many inline scripts
and need time to migrate:

```ts
// theo.config.ts
export default defineConfig({
  security: {
    headers: {
      cspMode: 'report-only', // ← back to report-only (default in 0.3.0 is 'enforce')
    },
  },
})
```

## Per-route gating

The Rails-inspired pattern lets you escalate specific routes to strict
mode WHILE keeping the global default at `warn`. This is the right
posture during a phased rollout — flip your auth endpoints first,
inventory the rest, then flip the global.

```ts
// theo.config.ts
export default defineConfig({
  security: {
    csrf: 'warn', // global stays in warn while migrating
    disallowedRoutes: ['/api/auth/*', /^\/api\/payments\//],
    disallowedBehavior: 'raise', // matched routes → 403 regardless of global
  },
})
```

`disallowedRoutes` accepts string globs (with `*` as wildcard) or
`RegExp` for finer control. `disallowedBehavior` is either `'warn'`
(still allow but flag in logs) or `'raise'` (403). See
[Known limitations](#known-limitations) for the exact-match gotcha.

## Gotchas

These come from real production cutovers (Rails, Next.js, Vue 3) and
from internal review against the enforcement playbook.

1. **`fetch` to a third-party API still triggers the warning if the URL
   is relative.** Use absolute URLs (`https://thirdparty.com/...`) or
   add the header anyway — it's harmless to the receiver.
2. **`navigator.sendBeacon` and `<form action=...>` posts cannot set
   custom headers.** Convert to `theoFetch` from a JS handler, or
   opt the receiving route out.
3. **WebSocket upgrades are not CSRF-checked.** The `Origin` header is
   already validated on upgrade by the framework, and there's no
   request body shape that CSRF could protect.
4. **The first request after each process restart re-emits each
   `csrf.warn` line.** `warnOnce` is process-scoped, not persistent.
   Plan your log sampling accordingly.
5. **CSP `nonce` MUST be cryptographically random per request.** The
   framework generates 16 bytes via `crypto.getRandomValues`. Do NOT
   reuse a nonce across requests, even within the same session.
6. **`Cache-Control: private, no-store` is auto-set on responses that
   carry a CSP nonce.** A CDN caching a nonce'd response defeats the
   point entirely.
7. **Inline scripts emitted by the framework itself (the SSR hydration
   data script) are auto-nonce'd.** Do not strip the auto-nonce — if
   it disappears, hydration breaks.
8. **`dangerouslySetInnerHTML` with a `<script>` payload is still
   blocked even if you set a nonce on the OUTER element.** Nonces
   apply per-tag, not per-tree.
9. **Strict mode adds ~5 µs of overhead per request.** Negligible in
   most apps; document for high-RPS workloads.

## FAQ

### My team can't migrate before our planned 0.3 bump. What do we do?

Pin to `theokit@0.2.x` until you're ready. The cutover is opt-in via
the `latest` dist-tag — staying on 0.2.x is a supported posture
through at least 0.4.0.

### Will `theokit@0.3.0` break my existing `theoFetch` calls?

No. `theoFetch` has attached `X-Theo-Action: '1'` since 0.2.0. If your
codebase uses `theoFetch` exclusively, the 0.3.0 bump is silent.

### Does CSP `enforce` mode break Tailwind?

No. Tailwind ships as a stylesheet bundle, not inline styles. The
default policy allows `style-src 'self'`. Inline `style="..."`
attributes on JSX elements are unaffected (the
`style-src-attr 'unsafe-inline'` directive is allowed by default).

### What about Vercel Analytics / Sentry / Plausible?

Each one ships an inline `<script>` you'll need to nonce or move. See
[Step 4](#4-audit-inline-scripts) for the mechanic.

### Can I roll back if the upgrade breaks production?

Yes. See [Rollback](#rollback).

### Does this affect WebSocket endpoints?

No. CSRF applies to HTTP state-mutating methods. WebSocket upgrade
already validates `Origin` separately, and the messages themselves
can't be CSRF'd from a third-party page (browsers gate cross-origin
WS over CORS).

### Why not auto-codemod the `fetch → theoFetch` swap?

Per ADR D1 in the cutover plan: the swap isn't always semantically
safe (some `fetch` calls intentionally bypass framework defaults).
A codemod would silently change behavior in those cases. Lint-then-
human-review is the right gate.

### Where do I file bugs?

GitHub issues at the theokit repo. Tag with `0.3-cutover` so the
maintainers see it on the dashboard.

## Rollback

If 0.3.0 ships and production breaks:

```bash
npm install theokit@0.2.0     # back to known-good
# or pin in package.json:
#   "theokit": "0.2.0"
```

`theokit@0.2.0` is supported on the `legacy` dist-tag indefinitely.
After rolling back, you have two options:

1. Stay on 0.2.x while you finish the migration; bump to 0.3 later.
2. Re-bump to 0.3.0 with the global escape hatches enabled
   (`csrf: 'warn'` + `cspMode: 'report-only'`), giving you 0.3.0's
   non-cutover features without the cutover risk.

## Known limitations

These are gaps documented at release-time. They are tracked for
future minor releases.

- **EC-13: `docsUrl` 404s until 0.4.0.** The `csrf.warn` payload links
  to `https://theokit.dev/upgrade/csrf-strict-cutover` which resolves
  to nothing until the hosted docs site ships. Fallback: read this
  file directly on GitHub or in `node_modules/theokit/docs/migrating/`.

- **EC-14: `disallowedRoutes` exact-string matching treats trailing
  slashes as distinct.** `'/api/login'` does NOT match `/api/login/`
  and vice-versa. Use RegExp for trailing-slash tolerance:
  `/^\/api\/login\/?$/`.

- **EC-17: version drift in this document.** The frontmatter carries
  `version: 0.3.0-beta.0`. If you're reading a cached copy, cross-
  reference the GitHub source for the latest. The truth source is
  `docs/migrating/0.2-to-0.3.md` on the main branch of the theokit
  repo at the time you're upgrading.

- **Scanner false-positive rate.**
  `theokit check --upgrade-readiness 0.3` is regex-based, not AST-
  based. It can produce false positives on heavily-templated source
  and false negatives on dynamic header construction
  (`headers[xVar] = '1'`). Treat its output as a starting point, not
  a complete audit. The grep commands in
  [Step 2](#2-audit-your-warn-logs) are the authoritative source —
  production logs don't lie.
