# 0015. Services runtime contract — Like-Vercel (fetch handler universal, file-system routing build-time, env runtime, structured stdout, conventional healthcheck)

* Status: accepted
* Date: 2026-05-27
* Accepted: 2026-05-27
* Deciders: [TheoKit owner]
* Tags: [architecture, runtime, polyglot, services, deploy, theocloud, like-vercel]

## Context and Problem Statement

[ADR-0012](./0012-mission-expansion-agent-products-on-like-vercel-runtime.md) expanded the mission with a literal phrase: "**eu achei que era só trocar o server**" (the owner's expectation: switching from local Node to Vercel to TheoCloud should be no more than changing an adapter — the service code stays the same).

[ADR-0014](./0014-services-as-external-processes.md) locked that services are external processes. This ADR locks the **shape** every external process — TheoKit-side and polyglot-side — must conform to, so:

1. **Local dev mirrors production.** A user testing with `docker-compose` sees the same request shape, env model, log format, healthcheck convention as the TheoCloud-deployed prod version will.
2. **Adapters are interchangeable.** The Vercel adapter, the CF Workers adapter, the AWS Lambda adapter, the (future) TheoCloud adapter and the local Node adapter all wrap the SAME contract.
3. **TheoCloud is provisioned before the adapter exists.** Validating the contract against a **TheoCloud-shaped harness** (docker-compose with caddy + structured-log collector + traceparent-aware proxy) lets the contract solidify before the adapter ships.

### This contract is global across the Theo product-mark, not internal to TheoKit

[ADR-0012](./0012-mission-expansion-agent-products-on-like-vercel-runtime.md) (invariant #4) establishes that **the moat is the cross-product standardization** — not any single product. This ADR is what makes that invariant concrete.

The Like-Vercel contract defined below is **enforced uniformly across three product surfaces**:

| Product surface | What the contract enforces here |
|---|---|
| `create-theokit` (absorbed scaffolder) | Templates generated by `--backend python` / `--backend node` MUST emit code that satisfies all 6 invariants below out of the box. A scaffolded project that fails the contract on day 0 is a bug. |
| TheoKit framework (`theokit dev` / `theokit build`) | The Like-Vercel contract is the shape `services: {}` orchestrates — proxy, healthcheck polling, traceparent injection, log routing all assume conformance. Services that violate the contract get loud errors at boot, not silent degradation. |
| TheoCloud (deploy target, adapter pending Wave 3) | The TheoCloud adapter, when it ships, MUST read `.theo/services.json` and host services that satisfy the contract verbatim. Adding a TheoCloud-only invariant (e.g., "TheoCloud also requires a `/cloud-health` endpoint with extra fields") would violate ADR-0012 invariant #4 and require BOTH ADRs amended in the same PR. |

**Consequence — what this ADR forbids by being global:**

- A future PR proposing "let's relax the healthcheck shape on TheoCloud because K8s liveness probes work differently" → REJECTED unless ADR-0012 invariant #4 is amended in the same PR
- A future PR proposing "let's add a 7th invariant that only applies to TheoCloud" → REJECTED unless the invariant works across all 3 surfaces or this ADR is amended to scope it
- A future PR proposing "let's allow `create-theokit --backend python` to generate templates that skip traceparent because uvicorn doesn't have it" → REJECTED; the template must include the middleware, not skip the invariant

**The contract is the moat. Any per-surface exception destroys it.**

## Decision Drivers

- **Owner statement "só trocar o server"** — the contract MUST make this literally true
- **Existing Vercel adapter (`packages/theo/src/adapters/vercel.ts:19-60`)** — already converts `(IncomingMessage, ServerResponse)` to Web `Request` and runs `executeRoute(request)`. The pattern is proven; we generalize it across services
- **Web Standards convergence** — Vercel Edge, CF Workers, Bun, Deno already use fetch handlers natively. Python (FastAPI ASGI) and Node (Hono fetch handler) can match
- **TheoCloud strategic priority** — provisioning is underway; contract must anticipate the adapter
- **Single mental model** — one set of conventions across 8+ deploy targets reduces docs / support / debugging surface

## Considered Alternatives

| Alternative | Rejected because |
|---|---|
| No contract — each adapter does its own thing | Path to chaos. Today's Vercel adapter shows how easy it is to diverge if not constrained. |
| Contract on Node-native IncomingMessage/ServerResponse (the legacy Express/Fastify shape) | Locks out CF Workers / Bun / Deno / Vercel Edge — all of which expose fetch handler natively. Forces shim layers everywhere. |
| Contract on Python WSGI | Async-incompatible. FastAPI is ASGI for a reason. WSGI doesn't fit Like-Vercel. |
| Contract on a custom RPC protocol (gRPC, GraphQL, etc.) | Locks users into one protocol surface. HTTP+OpenAPI is the lingua franca. |
| Define contract but allow per-adapter exceptions | Erodes "só trocar o server" — if Vercel adapter rejects requests CF Workers accepts, the contract has failed. |

## Decision

**The Like-Vercel runtime contract — 6 invariants every TheoKit service (TS or polyglot) MUST satisfy:**

### 1. Fetch handler is the universal entry

Every service exposes — directly or via a thin runtime adapter — a function of shape `(request: Request) => Promise<Response>` (Web Standards). Per language:

| Service runtime | How the contract is satisfied |
|---|---|
| TheoKit TS (in-tree) | `defineRoute` handler returns `Response`; framework's pipeline wraps it |
| Python (FastAPI) | ASGI app served by uvicorn. ASGI envelope is bijective with Web Request/Response (FastAPI does this implicitly). |
| Node sidecar (Hono) | `app.fetch: (Request) => Promise<Response>` — native |
| Future Node sidecar (Express/Fastify) | Wrap with `@hono/node-server` or equivalent — NOT in Wave 2 |

**The adapter's job is to translate the platform-native shape (Lambda Event, IncomingMessage, FetchEvent, ASGI scope) into a Web `Request` and pipe the `Response` back.** Vercel adapter at `packages/theo/src/adapters/vercel.ts:35-60` is the canonical reference.

### 2. File-system routing is BUILD-TIME

Routes are scanned ONCE at build time (`scanServerRoutes`) and baked into a manifest (`.theo/route-manifest.json` for TheoKit core, `services/<name>/openapi.json` for polyglot services). **No filesystem reads on the hot path.** Cold-start cache pattern (see `vercel.ts:30`) is mandatory for serverless adapters.

For polyglot services: each service emits its OpenAPI spec at build (or first-boot in dev) — TheoKit reads it to generate the typed client AND to validate the proxy contract.

### 3. Environment variables are RUNTIME, not build-time

`process.env.X` / `os.environ['X']` are read at cold start (or first access), NEVER bundled into the artifact. The Vercel adapter explicitly states this:

```ts
// packages/theo/src/adapters/vercel.ts:21
// Environment variables are resolved at RUNTIME, not build time.
```

Services follow the same rule. This is what allows TheoCloud (secrets manager) and Vercel (env UI) and local dev (`.env` file) to use the same code without rebuild.

### 4. Healthcheck is conventional and minimal

Every service MUST expose `GET /health` returning:

- `200 OK` with body `{ "status": "ok" }` when the service is ready to accept traffic
- `503 Service Unavailable` with body `{ "status": "starting" | "draining" | "unhealthy" }` otherwise

`pnpm dev` polls this endpoint before considering the service ready. Vercel/CF/TheoCloud deploy adapters consume the same convention. **No `/healthz` ambiguity, no `/api/health` vs `/health` debates.** One path, one shape.

### 5. Logs are structured JSON lines on stdout

Every service emits one JSON object per line to stdout, with at minimum:

```json
{
  "timestamp": "2026-05-27T10:00:00.000Z",
  "level": "info|warn|error|debug",
  "message": "...",
  "traceparent": "00-<trace-id>-<span-id>-01",
  "service": "agent-python"
}
```

No multi-line stack traces (escape them in `message` or `error` field). No ANSI colors in production. Local dev MAY render pretty — the wire format is JSON.

### 6. W3C Trace Context propagation is mandatory

When the proxy hop sends a request to a service:

- TheoKit injects `traceparent: 00-<traceId>-<spanId>-01` (already shipped in TheoKit core)
- Service reads `traceparent` from request headers
- Service logs MUST echo `traceparent` (see #5) so log correlation works
- Service MAY emit its own spans (OTel SDK in the service) — optional

This makes a single trace span the request from browser → TheoKit → service → response, across local dev, Vercel, and TheoCloud.

### Manifest — `.theo/services.json`

The build emits `.theo/services.json` to feed adapters:

```json
{
  "version": 1,
  "services": [
    {
      "name": "agent",
      "runtime": "python",
      "port": 8001,
      "proxy": "/api/agent",
      "openapi": "http://localhost:8001/openapi.json",
      "healthcheck": "/health",
      "dev": "uvicorn main:app --reload --port 8001",
      "build": null,
      "start": "uvicorn main:app --port 8001 --workers 4"
    }
  ]
}
```

Adapters consume this:

- **Local Node adapter:** generates docker-compose
- **Vercel adapter (Wave 3):** generates Vercel monorepo project with the Python service as a sibling Function
- **TheoCloud adapter (Wave 3):** generates K8s manifests (one Deployment + Service per entry)
- **CF Workers adapter:** rejects unsupported runtimes (no Python on CF Workers) with a clear error

### TheoCloud-shaped local harness (Wave 2 deliverable)

Until the TheoCloud adapter ships, local dev MUST use a docker-compose stack that mimics TheoCloud's shape:

- **Caddy in front** (single ingress; like TheoCloud's K8s Ingress)
- **Web container** (TheoKit TS app)
- **Service container(s)** (Python/Node services)
- **Postgres + Redis** (managed services in TheoCloud)
- **Env injection** via Docker secrets (mimics secrets manager)
- **Healthcheck depends_on conditions** (matches TheoCloud's readiness probes)
- **Structured logs** to stdout (collected by `docker compose logs` locally; would feed Loki/CloudWatch in prod)

This harness is the "ambiente parecido com o produtivo" the owner asked for. It validates the contract before TheoCloud is wired.

## Consequences

**Positive:**

- "Só trocar o server" becomes literally true — same code runs locally, on Vercel, on TheoCloud (when adapter ships)
- One mental model across 8+ deploy targets and 2+ service runtimes
- Trace correlation works out of the box across the proxy hop
- Adapters are diff-able by what they wrap, not by what they redefine
- Polyglot services drop into existing observability without bespoke wiring

**Negative:**

- Strict contract = some service patterns are out (e.g., Python services using WSGI + Flask without ASGI bridge — must use FastAPI or upgrade)
- Healthcheck convention pinned at `GET /health` — services with different convention (legacy `/healthz`, `/_status`) must change
- Structured-log requirement constrains library choices — services with hardcoded pretty-printing must be configured for JSON in prod

**Neutral:**

- The contract is a SHAPE, not a wire protocol. Services still own their internal architecture.
- Trace context propagation is mandatory but lightweight — single header.

## Implementation outline (not part of this decision; for sequencing only)

1. **`services` Zod schema** ([ADR-0014](./0014-services-as-external-processes.md) item) — encodes points 4–6 in the type
2. **Manifest emitter in `cli/commands/build.ts`** — writes `.theo/services.json` per spec above
3. **Python FastAPI template** — `services/agent-python/` with healthcheck + structlog (JSON renderer) + ASGI traceparent middleware
4. **Node Hono template** — `services/agent-node/` with `app.get('/health', ...)` + pino (JSON mode) + traceparent middleware
5. **Caddy + docker-compose generator** — generates the TheoCloud-shaped harness from the manifest
6. **Vite proxy plugin integration** — translates manifest into `server.proxy` for dev
7. **`theokit check --runtime-contract`** — validates a service against the contract (asserts healthcheck shape, log format sample, traceparent echo)

## Open questions (intentional — to resolve in Wave 2 design)

- Should `traceparent` be REQUIRED in service logs, or just RECOMMENDED? Probably required for production builds; warn in dev.
- Should `services.json` be versioned beyond `"version": 1`? Versioning policy as the spec grows.
- Should Caddy be the harness ingress, or Traefik, or nginx? Caddy chosen for simpler config; revisit if TheoCloud's actual ingress diverges.
