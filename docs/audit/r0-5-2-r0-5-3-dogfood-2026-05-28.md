## Plan-caused regressions: **ZERO**

The bugs surfaced during T1.2 local validation were **pre-existing** drift between the templates and the specs that nobody had caught because the specs were SKIP-gated. Each is fixed:

- Bug 1 — `drizzle.config.ts` CWD-relative paths — fixed in both postgres + saas configs
- Bug 2 — `template-postgres /api/users` returned `{ users }` instead of `[]` — fixed to align with api-only
- Bug 3 — `@usetheo/ui` missing from `template-saas` fixture deps — added + workspace-registered
- Bug 4 — `template-saas` spec used `username` field; route expects `email` — spec updated
- Bug 5 — 4 template fixtures NOT in `pnpm-workspace.yaml` — added (closes EC-2 from edge-case review)

These are pre-existing because the affected fixtures + specs lived in a SKIP-gated state where nobody saw them fail; the plan's "actually run them in CI" forced the latent issues to the surface. Classic dogfood value.

### Pre-existing template-default Playwright drift (3) — SDK migration

`fixtures/template-default/server/routes/chat.ts` was migrated to `@usetheo/sdk` `Agent.prompt` in commit `7e07053 feat: SDK v1.1.0 consumption — all 6 production-readiness primitives wired` (pre-dates this plan). The mock `Recebi: "..."` text the spec expects no longer exists. Three spec tests fail expecting it:

- `chat composer accepts input + auto-attaches X-Theo-Action header (T1.1)` → expects `Recebi: "..."` mock
- `streaming response arrives as 3 SSE events in order` → expects mock streaming behavior
- `no unhandled console errors during full session` → SDK errors when API key absent

**Plan-caused?** **No.** Verified via `git log fixtures/template-default/server/routes/chat.ts` — last modified by commit `7e07053` (pre-dates this plan by multiple iterations). The spec was written when the route had a synchronous Brazilian-Portuguese mock; the route is now SDK-driven and the spec is stale.

**Triage:** these 3 specs should be either (a) updated to use the canonical-chat project pattern (which sets `OPENROUTER_API_KEY=PLAYWRIGHT_PLACEHOLDER...` per `playwright.config.ts:168`) and assert SDK error events, or (b) split into a separate `template-default-mock` project with a route variant that retains the mock. Out of scope for this plan.
